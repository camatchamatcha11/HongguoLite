# AGENTS.md

本文件是 HongguoLite 项目级协作配置。后续实现任何模块时，先读本文件，再读 `./docs/HongguoLite_PRD.md` 和对应的 `./docs/Module*_Tasks.md`。

## 项目定位

HongguoLite 是对“红果短剧 App”的学习型 Android 复刻 Demo，不追求完整商业 App，而是围绕未来工作核心产品线训练：搜索、榜单、剧场列表和相关导航链路。

核心目标：

- 让 App 可运行、可演示、可讲清楚。
- 把 80% 精力放在搜索与榜单。
- 首页只做静态短剧播放页：背景图、顶部图标、右侧悬浮按钮、底部信息区。
- 商城、赚钱、我的只做最简占位，不扩展真实业务。

## 截图理解

红果短剧的主要布局架构如下：

- 底部 5 Tab：`首页 / 剧场 / 商城 / 赚钱 / 我的`，底部导航是全局框架。
- 首页：沉浸式短视频播放形态。竖屏全屏背景，上方有菜单和搜索，右侧有收藏、评论、点赞、分享，底部有剧名、标签、简介和“观看完整短剧”入口。本 Demo 只做静态版。
- 剧场页：顶部搜索框是核心入口，下面是横向频道 Tab，接着是筛选、排行榜、新剧、预约等入口，再往下是双列卡片流。
- 搜索前置页：返回键 + 搜索框 + 搜索按钮，快捷入口，猜你想搜，短剧热搜榜，必要时展示搜索历史。
- 搜索中态：输入后替换前置内容为联想列表。前几项可带封面，普通联想项只显示放大镜和文字，关键词用橙色高亮。
- 搜索结果页：顶部搜索栏保留关键词，下方横向分类 Tab，主体为双列卡片流，复用剧场卡片组件，并支持关键词高亮。
- 榜单页：顶部背景氛围较强，标题为“红果兴趣榜”，有返回和分享；主体是白色圆角面板，包含一级 Tab、二级 Tab/筛选入口和单列横向大卡片列表。
- 排行筛选弹层：从榜单页分类入口打开，底部/半屏圆角白色面板，多组筛选项，选中项用浅橙底和橙色文字。

注意：红果app的具体界面可以查看文件./hongguo app下的图片，并且只有当必要时采取查看图片从而完成每个模块的任务。

## Tier 优先级

- Tier 1：搜索前置页、搜索中态、搜索结果页、榜单页、剧场页“找剧”Tab。要求 UI 接近截图、交互完整、状态清晰、组件可复用。
- Tier 2：首页静态短剧播放页、剧场页其他频道。要求形态像、交互简化，可用 Toast 占位。
- Tier 3：商城、赚钱、我的。只做大图标和一行说明文字。

时间不够时，优先保住搜索三态和剧场到搜索入口。不要砍搜索三态。

## 技术栈约束

强制使用：

- Kotlin
- Jetpack Compose + Material3
- Navigation Compose
- MVVM：ViewModel + StateFlow
- Kotlin Coroutines + Flow
- Retrofit2 + OkHttp
- Coil 3
- DataStore Preferences，用于搜索历史
- 最低 SDK：API 24

禁止使用：

- Hilt / Dagger
- Room
- WorkManager
- 自定义协程框架
- 真实红果 API

## 工程风格

- 新建 Android 项目时使用包名 `com.example.hongguolite`。
- 使用 Gradle Kotlin DSL。
- 路由常量集中管理，不要在 UI 中散落魔法字符串。

- **禁止硬编码常量**：所有字符串、数字、尺寸、颜色等常量必须定义在 `app/src/main/res` 下（如 `strings.xml`、`dimens.xml`、`colors.xml`），或使用 Compose 的 `MaterialTheme` 与常量对象集中管理。代码逻辑块中不得直接出现字面量（如 `"提交"`、`16`、`"#FF6200EE"`）。

- Mock 数据先本地化，后续网络层再替换数据源。
- 业务数据模型要能覆盖剧场卡片、搜索结果、热搜榜、榜单卡片的共同字段。
- Compose 函数保持小而可读，重复 UI 抽成组件，例如 `DramaCard`、`RankCard`、`SearchBar`、`CategoryTabs`。
- 对于 `./docs/Module*_Tasks.md` 文件中的脚手架代码或示例代码，**禁止删除其中的注释**。这些注释是用户在学习过程中留下的实践记录，具有保留价值。

  在提供完整项目代码之后，你可以：

  - 对新增或修改的代码添加注释；
  - 对项目的重点、难点部分进行特别注释；
  - **为函数/方法添加注释，说明其作用、参数和返回值**，便于入门者理解逻辑。

  总之，注释有助于用户理解项目的实现思路，应优先保留和补充，而非移除。

## UI 实现准则

- 优先复刻布局关系和信息密度，不追求像素级间距。
- 移动端竖屏优先，目标设备以用户真机 vivo / iQOO 为主要验收环境。
- 列表使用稳定尺寸，避免图片加载或文字变化导致布局跳动。
- 卡片文字要限制行数并使用省略号。
- 搜索和榜单的橙色高亮、火焰热度、排名角标、胶囊标签是核心识别元素，要优先保留。
- 图标优先使用 Material Icons / Material Icons Extended。
- 首页、剧场、搜索、榜单之间的跳转要真实可走通，占位功能才用 Toast。

## 搜索模块规则

搜索是 Demo 核心。实现时必须覆盖：

- 前置页：默认词、快捷入口、猜你想搜、短剧热搜榜、搜索历史。
- 中态：输入非空后显示联想列表，300ms 防抖，支持旧请求/旧计算结果不覆盖新结果。
- 结果页：分类 Tab、双列结果列表、关键词高亮、清空按钮。
- 历史记录：DataStore 持久化，去重，最新搜索排最前，支持单条删除和清空。
- 状态表达：优先使用 `sealed class SearchState` 或同等清晰的状态模型。

## 榜单模块规则

榜单是另一条核心线。实现时必须覆盖：

- 顶部区域：返回、分享、标题、更新说明。
- 双层 Tab：一级 Tab 与二级 Tab 状态独立；一级切换时二级重置。
- 列表：`LazyColumn` + `RankCard`，单列横向大卡片，前 3 名排名角标有特殊颜色。
- 分类筛选：如实现弹层，使用半屏圆角白色面板，多组筛选项，选中态浅橙。
- Tab 切换后列表数据要变化，并尽量保持滚动流畅。

## 剧场与首页规则

- 剧场顶部搜索框点击必须跳转搜索前置页，不能只在剧场内输入。
- 剧场“排行榜”入口必须跳转榜单页。
- 剧场“找剧”Tab 使用双列卡片，并抽出可复用 `DramaCard`。
- 首页右上角搜索图标也要跳转搜索前置页。
- 首页不做视频播放、上下滑切换、红包、AI 声明、真实进度条。

## 调试与测试策略

- 开发 UI 模块时，优先拆小组件并为核心组件写 `@Preview`；复杂页面至少提供页面级 Preview 和关键组件 Preview。
- Preview 用于快速检查静态视觉、布局层级、文字溢出、不同状态和小屏宽度，避免每次改间距、字号、颜色都真机安装。
- 业务逻辑和状态变化优先写单元测试，例如 Tab 顺序、搜索历史去重、筛选状态、ViewModel 状态流，不依赖真机手点验证。
- 关键交互链路可写 Compose UI Test，例如点击搜索框触发跳转、点击排行榜入口、输入关键词后展示联想列表。
- 真机主要用于 Preview 和测试难以覆盖的部分：状态栏/安全区、系统字体、输入法、返回键、滚动手感、性能和真实设备观感。
- 成熟开发流程应是：Mock 数据 → 小组件 → Preview 调 UI → 页面组合 → 状态/导航接入 → 单元测试或 UI 测试 → 最后用户进行真机验收。

## 验收分工

Android 项目的验收分两层处理，以提高开发效率：

- Codex 负责代码级和构建级验证：读 PRD、写代码、解释关键实现、运行可用的 Gradle 编译/测试命令、根据报错日志定位问题。
- 用户负责 Android Studio 和真机验收：Sync、Run、连接手机、检查视觉效果、点击交互、返回键行为、滚动手感和截图对比。
- UI 子任务优先由用户在真机上验收，因为状态栏、安全区、字体、底部导航高度、卡片比例和滚动手感都更适合在目标设备上判断。
- 如果 Android Studio 或真机报错，用户提供错误截图、Build Output、Logcat 或现象描述，Codex 继续定位和修复。
- Codex 在每个子任务完成后，应明确列出用户需要手动验收的点，而不是把所有时间耗在命令行环境问题上。

## 协作约定

- 由于 Gradle 编译耗时很长，如果只是改 UI 和其它辅助功能改动，可以先做代码实现和局部检查，把完整 `assembleDebug` 留到关键节点再跑，从而加快响应速度。

- 后续 Codex 运行 Gradle 编译 / 测试命令时，应把日志通过命令行重定向写入项目根目录的 `build-log.txt`，例如使用 `.\gradlew.bat assembleDebug --console=plain --stacktrace *> build-log.txt`。命令结束后，Codex 只需总结关键错误、失败 task、warning 或成功结论，避免把完整构建日志直接贴进对话。

- 如果最后一次代码修改之后已经成功运行过相关 Gradle 验证命令，则真机验收前不需要为了“再确认一次”而重复运行同一构建命令；除非后续又改了代码、资源、Gradle 配置，或用户明确要求重新验证。

- 用户会逐步给出模块要求。不要提前实现后续模块，除非当前模块必须打通路由占位。
- 对新人学习价值高的地方，说明“为什么这么写”，但不要把最终代码写成教程腔。
- 遇到 PRD 与真实截图不一致时，先保留 PRD 功能，再用截图校准视觉。
- 每当协作方式、验收方式、需求范围或实现策略发生变化时，Codex 要主动询问用户是否需要把变动添加、修改或删除到 `AGENTS.md`、PRD、任务文档或其他合适文件中。未经用户确认，不要擅自把临时讨论写成长期规则。
