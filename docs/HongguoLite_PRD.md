## app特点分析

### 这个 App 的技术特点分析

我从截图里能读出的技术栈信号，按你将来工作涉及的优先级排序：

#### 1. 底部主 Tab 框架（接触概率 100%）

首页、剧场、商城、赚钱、我的——5个主 Tab。技术上叫 **BottomNavigationView + Fragment**。

------

#### 2. 沉浸式视频流（首页，接触概率低，但要理解）

首页那种"全屏视频 + 上下滑切换 + 右侧操作栏（收藏/评论/点赞/分享）"——这是客户端最复杂的场景之一，涉及：

- **ViewPager2 垂直滑动**
- **视频预加载**（下一个视频在当前视频还在播时就开始加载，保证切换丝滑）
- **视频复用**（不能每次滑动都 new 一个播放器，要有池化）
- **内存管理**（视频是内存大户，滑远了要释放）

------

#### 3. 双列瀑布流（剧场 & 搜索前置页，接触概率 90%）

剧场页和搜索页的"两列卡片"布局——这是客户端最常见的列表形态之一。技术点：

- **RecyclerView + GridLayoutManager**（或 StaggeredGridLayoutManager）
- **Adapter 与 ViewHolder** 模式
- **图片懒加载**（卡片里的封面图，用 Glide / Coil / 字节自研 Fresco 类库）
- **分页加载**（滑到底部自动加载下一页）
- **列表复用优化**（滑动不卡顿）

------

#### 4. 顶部 Tab + ViewPager（榜单、剧场，接触概率 95%）

榜单页的"预约榜/推荐榜/热播榜..."，剧场页的"找剧/社区/漫剧..."——都是这个模式。

技术点：

- **TabLayout + ViewPager2**
- 每个 Tab 对应一个 **Fragment**
- **懒加载**（切到某个 Tab 才初始化里面的数据，不然首次打开页面要加载 6 份数据，性能爆炸）
- Tab 切换时的**状态保持**（从榜单1切到榜单3再切回来，滑动位置是否保留？）

------

#### 5. 搜索功能（接触概率 100%，用户的项目组组核心业务）

从搜索页截图能看出搜索功能有这几层：

**(a) 搜索前置页**（用户还没输入时）

- 搜索框placeholder显示 "末世"（这叫**搜索词下发** / **默认词下发**，服务端根据用户画像推的）
- 快捷入口："识剧"（图片搜索）、"排行"、"上新"、"演员"
- "猜你想搜"模块（**个性化推荐**）
- "短剧热搜榜"（**全局热榜**）

**(b) 搜索中态**（用户正在输入）

- 从截图看不到，但标配是**搜索联想词** / **sug**（suggestion 的缩写，客户端术语）
- 技术点：输入防抖（debounce 300ms）、请求取消（避免旧结果覆盖新结果）

**(c) 搜索结果页**（用户点了"搜索"按钮）

- 从截图看不到，但标配是列表展示 + 可能有筛选器

## 1. 背景与目标

### 1.1 背景

学生即将入职字节跳动红果短剧 Android 客户端开发岗，组的产品方向是 **App 的搜索与榜单产品线**。Mentor 提到：剧场页的搜索框入口也归他们组维护。本 Demo 的目的不是做商业级产品，而是让学生在入职前**亲手实现一遍这条产品线最核心的交互和技术栈**。

### 1.2 目标

完成本 Demo 后，学生应该能够：

1. 独立搭建标准的 Compose Android 项目结构
2. **熟练实现搜索模块的三个状态**（前置页、中态联想、结果页），这是入职后最核心的工作
3. 实现复杂的双层 Tab 嵌套切换（榜单页）
4. 实现可复用的双列卡片组件（覆盖剧场、搜索结果、热搜榜等多场景）
5. 用 Retrofit + 协程 + Mock 数据演练真实的网络请求模式
6. 用 Coil 加载远程图片到列表
7. 100% 的代码自己写过、调试过、改过

### 1.3 非目标（明确不做的事）

- ❌ 视频播放（红果用字节自研播放器，入职后会接触）
- ❌ 用户登录、注册、支付
- ❌ 商城、赚钱、我的 Tab 的真实功能（仅做占位页）
- ❌ 真实接入红果 API（用 Mock 数据 + 公共开放 API 模拟）
- ❌ 像素级 UI 还原（接近形态即可，不抠每个间距字号）
- ❌ 运营元素：红包、新人券、金币 toast、AI 生成声明等
- ❌ 截图识别短剧、识剧、演员等高级功能（仅在 UI 上保留入口图标，点击 Toast 提示"未实现"）

## 2. 产品概述

### 2.1 产品名

**HongguoLite**（红果精简版）

### 2.2 保真度分层（核心设计原则）

不是所有页面都做到一样精细。按"是否对应学生将来的工作"分三层：

| 层级                                 | 模块                                                      | 实现要求                              |
| ------------------------------------ | --------------------------------------------------------- | ------------------------------------- |
| **Tier 1（核心，高保真+完整交互）**  | 搜索前置页、搜索中态、搜索结果页、榜单页、剧场页"找剧"Tab | UI 接近真实 App，交互完整，代码质量高 |
| **Tier 2（次要，UI 骨架+简化交互）** | 首页（短剧播放页静态版）、剧场页其他一级 Tab              | UI 看起来像，交互简化或 Toast 占位    |
| **Tier 3（占位）**                   | 商城 Tab、赚钱 Tab、我的 Tab                              | 一张大图标 + 一行说明文字             |

**这个分层是 Demo 设计的核心。** 学生需要把 80% 的精力压在 Tier 1 上。

### 2.3 信息架构


### Module 2：剧场页 [Tier 2 + Tier 1 部分]

**目标**：实现剧场页 UI 骨架，重点做好"找剧"子 Tab 的双列卡片，并打通"剧场搜索框 → 搜索前置页"的入口。

#### 2.1 顶部搜索框

UI 要求：

- 圆角矩形搜索框，左侧放大镜图标
- 占位文字："我在精神病院学斩神"（写死，作为热门词推荐效果）
- 占位文字右侧有红色【漫剧】小标签
- 右侧分隔线后："📷 截图识别短剧"图标 + 文字
- **整个搜索框点击 → 跳转到搜索前置页**（不能在剧场页本地输入）
- 截图识别图标点击 → Toast"截图识别未实现"

#### 2.2 一级 Tab（横向滚动）

7 个 Tab：找剧、社区、漫剧、电影、电视剧、听书、小说

实现要求：

- 默认选中"找剧"
- 选中态：黑色加粗 + 下方红色或橙色横线指示器
- 未选中态：灰色普通字
- **只有"找剧"Tab 下有内容（双列卡片），其他 Tab 切换过去显示一行文字"该分类未实现"**

技术：`ScrollableTabRow`

#### 2.3 4 个圆形功能入口

横向排列：筛选、排行榜、新剧、预约

实现要求：

- 圆形彩色背景（用不同颜色区分）+ 中间图标 + 下方文字
- **"排行榜"点击 → 跳转到榜单页**（Module 4 实现）
- 其他 3 个点击 → Toast"功能未实现"

#### 2.4 找剧 Tab 内容：双列卡片瀑布流 [Tier 1]

这是 Module 2 的核心，要做精。

卡片结构：

```
┌─────────────────────┐
│                     │
│      封面图          │  ← 占据卡片 ~85% 高度
│                     │
│  💧 4254万热度       │  ← 浮在封面右下角
└─────────────────────┘
  剧名（黑色加粗，最多 2 行省略）
  类型胶囊1  类型胶囊2     ← 灰底圆角胶囊
```

实现要求：

- 双列网格，每列卡片宽度约屏幕一半减边距
- 卡片左上角可能有红色【爆剧】或绿色【新剧】角标（数据驱动，不是所有卡片都有）
- 列表至少 20 条 Mock 数据
- 列表能流畅上下滚动
- 卡片**不需要**实现点击（点击只 Toast 即可，详情页本期不做）

技术栈：

- `LazyVerticalGrid` + `GridCells.Fixed(2)`
- 自定义可复用的 `DramaCard` Composable
- 占位封面图：暂用本地 Drawable，Module 5 替换为远程 URL

#### 2.5 验收标准

-  顶部搜索框样式正确，**点击能跳转到搜索前置页**
-  一级 Tab 横向滚动正常，能切换，"找剧"有内容，其他 Tab 显示占位文字
-  4 个圆形入口样式正确，"排行榜"能跳转
-  双列卡片至少 20 张，滚动流畅
-  卡片角标【爆剧】【新剧】数据驱动，部分有部分没有
-  `DramaCard` 抽成可复用组件（搜索结果页要复用）

### Module 3：搜索模块 [Tier 1，Demo 核心]

**目标**：实现搜索的三个状态，这是 Demo 中最核心、最能体现工程能力的部分，对应学生入职后的核心工作内容。

> **这个 Module 是整个 Demo 的重头戏。预估 2-3 天。**

#### 3.1 搜索前置页（用户刚进搜索页时）

**对应截图**：搜索前置页.jpg

UI 结构（从上到下）：

1. **顶部搜索栏**
   - 左侧返回箭头（点击返回上一页）
   - 中间圆角搜索框（自动获取焦点，可以输入）
   - 占位文字"我的26岁女房客"（写死的热门词推荐）
   - 右侧"搜索"按钮
2. **4 个圆形功能入口**：识剧、排行、上新、演员（参考剧场页的圆形入口样式，全部 Toast 占位）
3. **猜你想搜模块**
   - 标题"猜你想搜" + 右侧刷新图标 ↻（点击刷新数据，可以从 Mock 池里随机取）
   - **三列横向卡片**（注意是三列，不是双列）
   - 每张卡片：封面图 + 热度 + 剧名 + 类型标签
   - 部分卡片左上角有【爆剧】角标
4. **短剧热搜榜模块**
   - 标题"短剧" + 红色"热搜榜"（注意"热搜榜"是红色，"短剧"是黑色）
   - **三列横向卡片**，**左上角有大数字排名角标【1】【2】【3】**
   - 每张卡片：封面 + 热度 + 剧名 + 热搜值
   - 至少 6 张卡片（支持横向或纵向滚动展示更多）
5. **搜索历史区域**（如果有历史记录）
   - 在"猜你想搜"上方显示
   - 标题"最近搜索" + 右侧垃圾桶图标（清空全部）
   - 历史关键词以胶囊形式横向流式排列（用 `FlowRow`）
   - 每个胶囊点击 = 立即用该词搜索；长按 = 删除该条

技术挑战：

- **三列网格 vs 双列网格** —— 跟剧场页的双列不同，验证 `LazyVerticalGrid` 的灵活性
- **历史记录持久化** —— 用 `DataStore Preferences` 存储

#### 3.2 搜索中态（用户输入但未点搜索时）

**对应截图**：搜索中态.jpg

触发条件：搜索框获得焦点 + 输入框文字非空。

UI 结构：

- 顶部搜索框（继承前置页）
- 下方显示**联想列表**（替换掉"猜你想搜"和"热搜榜"）

联想列表两种条目类型：

**类型 A（带封面，前 N 条精确匹配的剧目）**：

- 左侧：放大镜图标 + 小封面图（80x80dp 左右）
- 右侧：剧名（关键词橙色高亮）+ 类型 + 热度

**类型 B（纯文字联想）**：

- 左侧：放大镜图标
- 文字（关键词橙色高亮），如"末世安沉"、"末世+重生+空间+囤物资+完结"

技术挑战（**这部分是 Demo 最有技术含量的地方**）：

1. **输入防抖（debounce）**
   - 用户连续输入时，不能每个字符都触发联想查询
   - 等用户停止输入 300ms 后才触发
   - 用 `Flow.debounce()` 实现
2. **关键词高亮**
   - 用 `AnnotatedString` 给关键词加 `SpanStyle(color = Orange)`
   - 处理大小写不敏感、关键词在文中多次出现等情况
3. **状态管理**
   - 三态切换：空（显示前置页内容）→ 输入中（显示联想）→ 已搜索（跳转结果页）
   - 用 `sealed class SearchState` 表达
4. **从 Mock 数据中模糊匹配**
   - 给定关键词"末"，从 30+ 条 Mock 数据中找出所有剧名包含"末"的剧目（带封面），再生成几条纯文字联想

#### 3.3 搜索结果页

**对应截图**：搜索结果.jpg

触发条件：用户在中态点击"搜索"按钮，或点击中态的某个联想项。

UI 结构：

1. 顶部搜索栏（搜索框内是已搜索的关键词，右侧有 X 清空按钮）
2. **结果分类 Tab（横滚）**：综合、漫剧、社区、影视、小说、听书、用户
   - "综合"默认选中
   - 切换 Tab 显示不同分类的结果（用同一份 Mock 数据按 `category` 字段过滤即可，无需做真实分类）
3. **双列卡片列表**：复用 Module 2 的 `DramaCard` 组件
   - 关键差异：**剧名中关键词需要橙色高亮**

技术挑战：

- **复用 `DramaCard` 但支持高亮** —— 给 `DramaCard` 加一个可选参数 `highlightKeyword: String? = null`
- **从历史搜索点回填** —— 历史胶囊点击后，搜索框文字变成该词 + 直接进入结果页

#### 3.4 验收标准（Module 3）

**搜索前置页：**

-  4 个圆形入口、猜你想搜（3 列）、热搜榜（3 列带排名）UI 完整
-  历史记录正常显示，可单条删除、可清空、可点击搜索
-  历史记录在 App 重启后**仍然存在**（DataStore 持久化生效）
-  同一关键词重复搜索不会在历史中重复，最新的排在最前

**搜索中态：**

-  输入文字时实时显示联想列表
-  实现了 300ms 防抖（连续打字不会卡，停止打字才触发联想）
-  关键词在联想结果中橙色高亮
-  联想列表中既有带封面的剧目（类型 A），也有纯文字（类型 B）

**搜索结果页：**

-  顶部分类 Tab 可切换，不同 Tab 显示不同结果
-  双列卡片复用了 `DramaCard` 组件
-  结果中关键词橙色高亮
-  搜索框内有清空按钮，点击清空后回到中态/前置页

**整体：**

-  三态切换流畅，无卡顿、无闪烁
-  至少 5 次 Git commit，每个 commit 信息清晰

### Module 4：榜单页 [Tier 1]

**目标**：实现复杂的双层 Tab 嵌套切换 + 单列横向大卡片，覆盖学生将来工作的另一核心场景。

#### 4.1 顶部区域

- 左上角返回箭头
- 右上角分享图标（点击 Toast）
- 标题"〘红果兴趣榜〙"（用艺术字符 `〘 〙` 装饰）
- 副标题"4月28日已更新·基于红果观看/互动以及个人兴趣排序 ❓"

#### 4.2 双层 Tab（核心难点）

**第一层（黑色文字 Tab）**：全部、漫剧、AI短剧、演员

- 选中态：黑色加粗 + 字号略大
- 未选中态：灰色

- **两层 Tab 状态联动** —— 一级 Tab 切换时，二级 Tab 应该重置或保持？

- 用 `ViewModel` 的 `StateFlow` 管理 `selectedTopTab` 和 `selectedSubTab` 两个状态

### Module 5：网络层接入（Retrofit + 协程）

**目标**：把前面 Module 的 Mock 数据全部替换成真实网络请求，演练完整的网络请求流程。

#### 5.1 数据层架构

要求采用 Repository 模式：

```
UI (Composable)
    ↓
ViewModel (StateFlow<UiState>)
    ↓
Repository (interface)
    ↓
RemoteDataSource (Retrofit) / MockDataSource (本地)
```

**抽象 `Repository` 接口**，方便切换实现。Module 2-4 用 `MockRepository`，Module 5 加上 `RemoteRepository`。

#### 5.2 接入开放 API

候选 API（任选其一即可）：

- [JsonPlaceholder](https://jsonplaceholder.typicode.com)：经典练习用 API
- [豆瓣电影 API 镜像](https://github.com/jdkanani/douban-api)（可能不稳定）
- 其他公共 RESTful API

请求要求：

- 至少 1 个真实 API 调用，把响应数据转换成剧目卡片显示
- 用 `suspend` 函数 + `viewModelScope.launch` 发起
- Loading / Success / Error 三态用 `sealed class UiState` 表达

#### 5.3 网络配置

- 添加网络权限：`<uses-permission android:name="android.permission.INTERNET" />`
- 配置 OkHttp 日志拦截器（Debug 包打印请求/响应日志）
- 处理 HTTPS 证书（如果 API 是 HTTP，需要 `usesCleartextTraffic="true"`）

#### 5.4 验收标准（Module 5）

-  至少一个页面（推荐：剧场找剧 Tab）的数据来自真实网络请求
-  Loading 时显示进度条
-  网络失败时显示错误页 + 重试按钮
-  OkHttp 日志在 Logcat 中能看到完整请求/响应
-  Repository 接口已抽象，Mock 和 Remote 实现都存在
