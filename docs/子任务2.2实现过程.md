# 子任务 2.2 实现过程：剧场页顶部

> 文件名按你的要求保留为“子任务1.2实现过程.md”，但本文实际讲的是 `Module2_Tasks.md` 中的“子任务 2.2：剧场页顶部”。

## 1. 先明确这次只做什么

这次没有继续做子任务 2.1 的 Mock 剧目数据，也没有提前实现 2.3/2.4 的 `DramaCard` 和双列列表。

子任务 2.2 只负责剧场页顶部区域：

- 顶部搜索框
- “截图识别短剧”入口
- 横向频道 Tab：找剧 / 看剧 / 漫画 / 电影 / 电视剧 / 听书 / 小说
- 四个快捷入口：筛选 / 排行榜 / 新剧 / 预约
- 将剧场底部 Tab 从占位页接入真正的 `TheaterScreen`
- 增加 `@Preview`，方便在 Android Studio 中直接预览 UI

这个拆法很重要：一个模块不要一次性全做完。先把顶部骨架打通，后续列表、卡片、搜索、榜单可以逐步接进来。

## 2. 第一步：抽出频道 Tab 数据

新增文件：

```text
HongguoLite/app/src/main/java/com/example/hongguolite/data/model/TheaterTab.kt
```

核心代码：

```kotlin
data class TheaterTab(
    val id: String,
    val label: String
)

val theaterTabs = listOf(
    TheaterTab(id = "find_drama", label = "找剧"),
    TheaterTab(id = "community", label = "看剧"),
    TheaterTab(id = "manga", label = "漫画"),
    TheaterTab(id = "movie", label = "电影"),
    TheaterTab(id = "tv", label = "电视剧"),
    TheaterTab(id = "audiobook", label = "听书"),
    TheaterTab(id = "novel", label = "小说"),
)
```

这里没有把 Tab 文案直接写死在 UI 里，而是抽成 `theaterTabs`。

这样做的好处：

- UI 只负责展示，不负责保存业务配置。
- 后续如果调整 Tab 顺序，只改一个列表。
- `id` 可以给后续业务逻辑使用，比如根据 `find_drama` 加载“找剧”列表。

## 3. 第二步：新增剧场顶部组件

新增文件：

```text
HongguoLite/app/src/main/java/com/example/hongguolite/ui/screens/theater/TheaterTopBar.kt
```

我把顶部区域拆成了几个小 Composable：

```kotlin
@Composable
fun TheaterTopBar(...) {
    Column {
        TheaterSearchRow(...)
        TheaterChannelTabs(...)
        TheaterQuickEntryRow(...)
    }
}
```

这是一种常见的 Compose 写法：外层组件负责组织结构，内部小组件分别负责具体区域。

## 4. 搜索框实现思路

核心代码：

```kotlin
Row(
    modifier = Modifier
        .fillMaxWidth()
        .padding(horizontal = 14.dp, vertical = 8.dp)
        .height(46.dp)
        .clip(RoundedCornerShape(14.dp))
        .background(Color.White.copy(alpha = 0.96f)),
    verticalAlignment = Alignment.CenterVertically
) {
    Row(
        modifier = Modifier
            .weight(1f)
            .clickable(onClick = onSearchBoxClick),
        verticalAlignment = Alignment.CenterVertically
    ) {
        Icon(imageVector = Icons.Default.Search, contentDescription = "搜索")
        Text(text = "我在精神病院学斩神", maxLines = 1)
    }

    Box(
        modifier = Modifier
            .height(24.dp)
            .width(1.dp)
            .background(Color(0xFFE8E8E8))
    )

    Row(
        modifier = Modifier.clickable(onClick = onScreenshotClick)
    ) {
        Icon(imageVector = Icons.Default.CameraAlt, contentDescription = "截图识别短剧")
        Text(text = "截图识别短剧")
    }
}
```

这里的关键点：

- 外层 `Row` 做成白色圆角搜索条。
- 左侧搜索区域使用 `Modifier.weight(1f)`，让它占据剩余空间。
- 中间用一个很细的 `Box` 模拟分割线。
- 右侧截图识别入口单独设置点击事件。

为什么不直接用 `TextField`？

因为剧场页顶部的搜索框只是入口，点击后跳转搜索页，不需要在这里输入。因此用普通 `Row + clickable` 更简单，也更符合当前任务范围。

## 5. 横向频道 Tab 实现思路

核心代码：

```kotlin
ScrollableTabRow(
    selectedTabIndex = safeSelectedIndex,
    containerColor = Color.Transparent,
    edgePadding = 14.dp,
    divider = {},
    indicator = { tabPositions ->
        TabRowDefaults.SecondaryIndicator(
            modifier = Modifier
                .tabIndicatorOffset(tabPositions[safeSelectedIndex])
                .padding(horizontal = 20.dp),
            height = 3.dp,
            color = TheaterOrange
        )
    }
) {
    theaterTabs.forEachIndexed { index, tab ->
        val selected = index == selectedTabIndex

        Tab(
            selected = selected,
            onClick = { onTabSelected(index) },
            text = {
                Text(
                    text = tab.label,
                    fontSize = 26.sp,
                    fontWeight = if (selected) FontWeight.Bold else FontWeight.Medium
                )
            }
        )
    }
}
```

这里选择 `ScrollableTabRow`，不是普通 `TabRow`。

原因是剧场顶部有 7 个频道，手机宽度放不下。用 `ScrollableTabRow` 可以横向滑动，和截图中的红果剧场页更接近。

这里还有一个小保护：

```kotlin
val safeSelectedIndex = selectedTabIndex.coerceIn(theaterTabs.indices)
```

它的作用是避免 `selectedTabIndex` 越界。虽然当前状态正常情况下不会越界，但 UI 组件尽量对输入参数宽容一点。

## 6. 四个快捷入口实现思路

核心代码：

```kotlin
Row(
    modifier = Modifier
        .fillMaxWidth()
        .padding(horizontal = 14.dp, vertical = 10.dp),
    horizontalArrangement = Arrangement.spacedBy(10.dp)
) {
    TheaterQuickEntry(Icons.Default.FilterList, "筛选", ...)
    TheaterQuickEntry(Icons.Default.LocalFireDepartment, "排行榜", ...)
    TheaterQuickEntry(Icons.Default.PlayArrow, "新剧", ...)
    TheaterQuickEntry(Icons.Default.Apps, "预约", ...)
}
```

每个入口复用同一个小组件：

```kotlin
@Composable
private fun TheaterQuickEntry(
    icon: ImageVector,
    label: String,
    iconBackground: Color,
    onClick: () -> Unit,
    modifier: Modifier = Modifier,
) {
    Row(
        modifier = modifier
            .height(54.dp)
            .clip(RoundedCornerShape(14.dp))
            .background(Color.White)
            .clickable(onClick = onClick)
    ) {
        Icon(...)
        Text(text = label)
    }
}
```

这里的设计重点是“重复 UI 抽组件”。

如果四个入口都手写一遍，代码会很长，而且后续调圆角、字体、图标大小时要改四处。抽成 `TheaterQuickEntry` 后，只需要改一处。

## 7. 第三步：新增 TheaterScreen 作为页面容器

新增文件：

```text
HongguoLite/app/src/main/java/com/example/hongguolite/ui/screens/theater/TheaterScreen.kt
```

核心代码：

```kotlin
@Composable
fun TheaterScreen(
    onSearchBoxClick: () -> Unit,
    onRankClick: () -> Unit,
    modifier: Modifier = Modifier,
) {
    val context = LocalContext.current
    var selectedTabIndex by remember { mutableIntStateOf(0) }

    Column(
        modifier = modifier
            .fillMaxSize()
            .background(Color(0xFFF7F7F7))
    ) {
        TheaterTopBar(
            selectedTabIndex = selectedTabIndex,
            onTabSelected = { selectedTabIndex = it },
            onSearchBoxClick = onSearchBoxClick,
            onRankClick = onRankClick,
            onFilterClick = {
                Toast.makeText(context, "筛选暂未实现", Toast.LENGTH_SHORT).show()
            },
            ...
        )

        TheaterTabPlaceholder(...)
    }
}
```

这里做了两件事：

1. 用 `selectedTabIndex` 保存当前选中的频道。
2. 把点击事件传给 `TheaterTopBar`。

为什么 `TheaterTopBar` 不自己保存选中状态？

因为顶部只是展示组件，页面才应该决定当前选中哪个 Tab。这样后续接 ViewModel 时会更自然：

```kotlin
TheaterUiState(
    selectedTabIndex = 0,
    dramas = ...
)
```

当前为了简单，先用 `remember { mutableIntStateOf(0) }`。等模块变复杂后，再迁移到 ViewModel。

## 8. 第四步：把剧场路由接到真实页面

修改文件：

```text
HongguoLite/app/src/main/java/com/example/hongguolite/ui/MainScreen.kt
```

原来剧场 Tab 是占位页：

```kotlin
composable(Routes.THEATER) {
    PlaceholderScreen(...)
}
```

现在改成：

```kotlin
composable(Routes.THEATER) {
    TheaterScreen(
        onSearchBoxClick = {
            // Module 3 接入后跳转搜索前置页。
        },
        onRankClick = {
            // Module 4 接入后跳转榜单页。
        }
    )
}
```

这里没有提前实现搜索页和榜单页跳转，因为它们属于后续模块。

但是接口已经预留好了：

- `onSearchBoxClick`：后续接 `navController.navigate(Routes.SEARCH)`
- `onRankClick`：后续接 `navController.navigate(Routes.RANK)`

这就是“先留接口，后接功能”的开发方式。

## 10. 第六步：写一个很小的单元测试

新增文件：


核心代码：

```kotlin
class TheaterTabTest {

    @Test
    fun theaterTabs_keepExpectedOrder() {
        assertEquals(
            listOf("找剧", "看剧", "漫画", "电影", "电视剧", "听书", "小说"),
            theaterTabs.map { it.label }
        )
    }

```

这个测试不是为了测试 UI 漂不漂亮，而是锁定“频道顺序”。

频道顺序属于产品结构，不应该被无意中改乱。测试能帮你在以后重构时发现这种问题。

## 11. 验证方式

我当时跑了两类验证：

```powershell
.\gradlew.bat testDebugUnitTest --tests com.example.hongguolite.data.model.TheaterTabTest
```

作用：验证频道数据测试通过。

```powershell
.\gradlew.bat assembleDebug
```

作用：验证 Android debug 包可以正常编译。

注意：这两个命令只能说明代码和构建层面没问题，不能代替真机验收。

真机仍然需要你重点看：

- 顶部搜索框是否贴近截图
- 横向 Tab 是否能滑动
- 四个快捷入口是否拥挤
- vivo / iQOO 上字体和底部导航是否协调
- 点击入口是否符合预期

## 12. 这次开发的核心思路

这次不是一上来直接写完整剧场页，而是按模块边界逐步推进：

1. 先抽数据配置：`TheaterTab.kt`
2. 再写顶部组件：`TheaterTopBar.kt`
3. 把顶部组件拆成搜索栏、频道 Tab、快捷入口三个区域
4. 新建页面容器：`TheaterScreen.kt`
5. 在页面里管理当前 Tab 状态
6. 接入主导航：`MainScreen.kt`
7. 给组件和页面都写 `@Preview`
8. 写小测试锁定关键配置
9. 最后跑构建验证

这个流程适合初学者反复练习。

你以后做一个新模块时，也可以照这个顺序：

- 先明确这次只做什么，不做什么。
- 先定义数据结构。
- 再做可复用 UI 小组件。
- 用 Preview 快速调小组件。
- 最后组合成页面。
- 页面接入导航。
- 核心配置或业务逻辑补测试。
- 真机只做最终体验验收，不要每改一点都装包。
