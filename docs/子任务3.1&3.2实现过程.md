# 子任务 3.1 & 3.2 实现过程

本文档记录 Module 3 中“子任务 3.1：架构理解 + 路由接入”和“子任务 3.2：Repository 抽象”的实现过程。重点不是复述完整代码，而是梳理一个项目子模块从任务文档到可运行代码的开发路径。

这两个子任务是搜索模块的地基：

- 3.1 负责让 App 里真正存在一个“搜索页路由”，并让首页和剧场页能跳过去。
- 3.2 负责让搜索模块先有一层数据抽象，后续 ViewModel 和 UI 不直接依赖 Mock 数据。

## 1. 开发前先判断任务边界

实现前先读了三类资料：

1. `AGENTS.md`：确认项目协作规则、工程风格、注释要求、测试和验收方式。
2. `docs/HongguoLite_PRD.md`：确认搜索是核心模块，首页和剧场都需要能进入搜索。
3. `docs/Module3_Tasks.md`：确认当前只做 3.1 和 3.2，不提前实现搜索三态完整 UI。

这里最重要的判断是：3.1 和 3.2 不是“把搜索页面全做完”，而是先把路由和数据层骨架搭出来。

所以本轮没有实现：

- 搜索输入联想 UI。
- 搜索结果双列列表。
- DataStore 持久化历史记录。
- SearchViewModel 的 300ms debounce。

这些属于后续 3.3 到 3.7。

## 2. 为什么先写测试

这次采用了测试先行的方式。原因是 3.1 和 3.2 虽然看起来只是“加几个文件”，但它们决定了后续模块依赖的接口形状。

先写测试可以提前固定两个关键约定：

- 路由里必须有统一的 `Routes.SEARCH`，避免 UI 中散落 `"search"` 字符串。
- `MockSearchRepository` 必须支持搜索、联想词、热搜、猜你喜欢和历史记录的基本行为。

新增的测试文件是：

- `app/src/test/java/com/example/hongguolite/navigation/RoutesTest.kt`
- `app/src/test/java/com/example/hongguolite/data/repository/MockSearchRepositoryTest.kt`

测试先失败是正常现象。当时失败点正好说明目标代码还不存在：

- `Routes.SEARCH` 未定义。
- `MockSearchRepository` 未定义。

这一步的意义是让“要做什么”变成可以自动检查的规则，而不是只靠人工记忆。

## 3. 子任务 3.1：路由接入

### 3.1.1 新增搜索路由常量

项目已有 `Routes.kt` 集中管理路由：

```kotlin
object Routes {
    const val HOME = "home"
    const val THEATER = "theater"
    const val SHOP = "shop"
    const val EARN = "earn"
    const val PROFILE = "profile"
    const val SEARCH = "search"
}
```

这样写的原因是：路由字符串属于全局导航协议，不能散落在不同 UI 文件里。否则后续如果改路由名，需要全项目搜索字符串，容易漏改。

`SEARCH` 不是底部 Tab，它是从首页或剧场进入的子页面。

### 3.1.2 新建搜索占位页

新增了 `SearchScreen.kt`。

这个页面现在只是占位壳：

- 顶部返回按钮。
- 搜索框外观。
- “搜索”按钮文字。
- 中间占位文案。

为什么先做占位页？

因为 3.1 的目标是验证导航链路。只要首页和剧场能跳到一个真实 Composable，就说明路由接入已经完成。完整搜索 UI 放到 3.4、3.5、3.6 再做，模块边界更清楚。

`SearchScreen` 接收一个参数：

```kotlin
fun SearchScreen(
    onBackClick: () -> Unit,
    modifier: Modifier = Modifier,
)
```

这里没有直接拿 `NavController`，而是把“返回事件”作为函数传进来。这样页面本身只关心“用户点了返回”，不关心“具体怎么退栈”。这是一种常见的 Compose 页面解耦方式。

### 3.1.3 在 MainScreen 中注册路由

`MainScreen.kt` 是全局导航入口。搜索页在 `NavHost` 里注册：

```kotlin
composable(Routes.SEARCH) {
    SearchScreen(
        onBackClick = {
            navController.popBackStack()
        },
    )
}
```

`popBackStack()` 的效果是：从哪里进入搜索，就返回哪里。

- 从首页进入，返回首页。
- 从剧场进入，返回剧场。

这比硬编码 `navigate(Routes.HOME)` 更合理，因为搜索入口不止一个。

### 3.1.4 首页和剧场入口跳转搜索页

首页本来已经把搜索按钮事件暴露成参数：

```kotlin
HomeScreen(
    onSearchClick = {
        navController.navigate(Routes.SEARCH)
    },
)
```

剧场页顶部搜索框同理：

```kotlin
TheaterScreen(
    onSearchBoxClick = {
        navController.navigate(Routes.SEARCH)
    },
)
```

这里的设计重点是：`HomeScreen` 和 `TheaterScreen` 不直接知道路由字符串，它们只把点击事件交给外层。真正导航由 `MainScreen` 统一处理。

这样页面更容易预览、复用和测试。

### 3.1.5 搜索页隐藏底部导航

项目底部有 5 个主 Tab，但搜索页不是主 Tab。实现方式是根据当前路由判断是否显示底部导航：

```kotlin
val shouldShowBottomBar = bottomTabs.any { it.route == currentRoute }
```

只有当前路由属于 `bottomTabs` 时才显示底部导航。`Routes.SEARCH` 不在 `bottomTabs` 中，所以搜索页打开时底部导航自动隐藏。

这比写 `currentRoute != Routes.SEARCH` 更可扩展。后续如果新增榜单详情页、短剧详情页，只要它们不是底部 Tab，底部导航也会自然隐藏。

## 4. 子任务 3.2：Repository 抽象

### 4.1 为什么需要 Repository

如果 ViewModel 或 UI 直接读取 `MockDramas.theaterList`，短期能跑，但后续会有几个问题：

- UI 会知道 Mock 数据来自哪里，层级混乱。
- 后续接 Retrofit 时，需要改动 UI 或 ViewModel 里很多地方。
- 搜索历史、联想词、热搜榜这些数据来源不一定相同，直接读 Mock 列表会变得难维护。

所以 3.2 先加一层 `SearchRepository`：

```text
UI
  -> ViewModel
      -> SearchRepository
          -> MockSearchRepository
```

后续 Module 5 接网络时，可以把实现替换成 `RemoteSearchRepository` 或组合多个数据源，但 ViewModel 不需要大改。

### 4.2 SearchRepository 接口设计

接口放在：

`app/src/main/java/com/example/hongguolite/data/repository/SearchRepository.kt`

它定义搜索模块目前需要的数据能力：

- `searchDramas(keyword)`：根据关键词搜索短剧。
- `getHotSearches()`：搜索前置页的热搜榜数据。
- `getGuessLikes()`：搜索前置页的猜你想搜数据。
- `getSuggestions(keyword)`：输入过程中的联想词。
- `getSearchHistory()`：搜索历史流。
- `addSearchHistory(keyword)`：添加历史。
- `removeSearchHistory(keyword)`：删除单条历史。
- `clearSearchHistory()`：清空历史。

为什么搜索、热搜、联想这些方法用 `suspend fun`？

因为现在虽然只是本地 Mock 数据，但未来很可能变成网络请求或数据库读取。提前使用 `suspend`，后续替换数据源时 ViewModel 调用方式不用变。

为什么搜索历史用 `Flow<List<String>>`？

因为历史记录会变化。用户新增、删除、清空历史后，UI 应该自动更新。`Flow` 很适合表达这种“持续观察的数据”。

## 5. MockSearchRepository 的实现思路

`MockSearchRepository` 放在：

`app/src/main/java/com/example/hongguolite/data/repository/MockSearchRepository.kt`

它暂时基于 `MockDramas.theaterList` 工作。

### 5.1 搜索逻辑

搜索时先 trim 关键词：

```kotlin
val normalizedKeyword = keyword.trim()
```

如果关键词为空，直接返回空列表。这样可以避免空关键词把所有短剧都搜出来。

匹配范围包括：

- `drama.title`
- `drama.tags`

这样做是为了复用已有 `Drama` 模型，不急着新建搜索专用模型。当前阶段更重要的是先把搜索模块链路跑通。

### 5.2 热搜和猜你想搜

当前实现使用 Mock 列表的前几条：

```kotlin
getHotSearches() -> take(10)
getGuessLikes() -> take(8)
```

这不是最终产品逻辑，而是模块开发中的临时数据策略。它能保证后续 UI 有稳定数据可以展示。

### 5.3 联想词逻辑

联想词从标题和标签里提取：

```kotlin
drama.title + drama.tags
```

然后按关键词过滤、去重、限制数量。

去重很重要，因为多个短剧可能有同一个标签。如果不去重，联想列表里会出现很多重复项，新手调 UI 时容易误以为是列表渲染出了问题。

### 5.4 搜索历史逻辑

3.2 阶段先用内存版历史记录：

```kotlin
MutableStateFlow<List<String>>(emptyList())
```

添加历史时遵循三个规则：

1. 去掉前后空格。
2. 空字符串不记录。
3. 重复搜索时移到最前面，而不是生成重复项。

这和常见移动 App 搜索历史行为一致。

注意：这个历史记录现在 App 进程结束就会丢失。Module 3.7 会把它替换成 DataStore 持久化。

## 6. 为什么代码里要补注释

这个项目是学习型项目，所以注释不是装饰，而是学习路径的一部分。

本轮补注释时遵循了一个原则：注释解释“代码表面看不出来的设计意图”，不解释显而易见的语法。

例如应该注释：

- 为什么搜索页不是底部 Tab。
- 为什么 `SearchScreen` 不直接拿 `NavController`。
- 为什么 Repository 方法提前用 `suspend`。
- 为什么历史记录用 `Flow`。
- 为什么 Mock 历史后续会被 DataStore 替换。

不需要注释：

- `fontSize = 14.sp` 是设置字体大小。
- `Modifier.padding(...)` 是设置间距。
- `Text(...)` 是显示文字。

这种注释密度更适合新人阅读：重点解释架构和数据流，而不是把代码逐行翻译成中文。

## 8. 这一轮学到的开发顺序

对于一个新模块，不要一上来就写完整 UI。更稳的顺序是：

1. 先读任务文档，明确当前子任务边界。
2. 先把路由或数据接口这种“地基”搭好。
3. 写测试固定核心行为。
4. 做最小可运行页面，占住导航位置。
5. 抽 Repository，避免 UI 直接依赖 Mock 数据。
6. 补关键注释，解释设计意图。
7. 跑测试确认行为没有破。
8. 再进入后续 ViewModel 和 UI 状态开发。

这也是 Module 3 后续任务的基础：

- 3.3 可以基于 `SearchRepository` 写 `SearchViewModel`。
- 3.4 可以把 `SearchScreen` 占位内容替换成搜索前置页。
- 3.5 可以接入输入联想和关键词高亮。
- 3.6 可以复用 Repository 搜索结果实现结果页。
- 3.7 可以把内存历史替换为 DataStore。

## 10. 给初学者的理解重点

你可以把 3.1 和 3.2 理解成“先修路，再准备仓库”。

3.1 修的是页面之间的路：

```text
首页 / 剧场
    -> MainScreen 统一处理点击
        -> Routes.SEARCH
            -> SearchScreen
```

3.2 准备的是搜索模块的数据仓库：

```text
未来 SearchViewModel
    -> SearchRepository 接口
        -> 现在用 MockSearchRepository
        -> 以后可换成 Remote/DataStore 实现
```

这样开发的好处是，每一步都能单独验证，也方便后续模块逐步替换，而不是把 UI、数据、导航全部混在一个文件里。
